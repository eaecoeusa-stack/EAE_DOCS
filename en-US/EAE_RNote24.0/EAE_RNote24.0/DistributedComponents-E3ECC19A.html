Compatibility of EcoStruxure Automation Expert version 24.0 components

Use EcoStruxure Automation Expert - Buildtime version 24.0 only when configuring: 

  • a Modicon M251 Distributed PAC with firmware version 24.0 

  • a Modicon M262 Distributed PAC with firmware version 24.0 

  • a Modicon M580 Distributed PAC with firmware version 24.0 

  • an Altivar Distributed PAC module with firmware version 24.0 

  • an EcoStruxure Automation Expert - Soft dPAC with version 24.0 

  • an EcoStruxure Automation Expert - Soft dPAC HA with version 24.0 

  • a CRD remote IO with firmware version 24.0 

  • an EcoStruxure Automation Expert - HMI with version 24.0 

  • an EcoStruxure Automation Expert - Archive with version 24.0 

EcoStruxure Automation Expert - Buildtime version 24.0 cannot be used to configure components having a version different than 24.0. 

EcoStruxure Automation Expert components version 24.0 cannot be used to communicate with components having a version different than 24.0.

Incorrect 61499 behavior

Incorrect 61499 behavior with unassociated adapter data being exposed out of the BasicFB has been fixed. 

If using an adapter in a basic function blocks in EcoStruxure Automation Expert up to and including v23.1, the change of an adapter's input variable from the basic's algorithm was passed to the destination adapter immediately instead of using the usual 1499 data queuing. In some applications this resulted in data being overwritten before being processed or data being sent that were not valid at the time of the event.  

Since v24.0 the data transferred between the adapters respects the WITH construct and is enqueued with the values at the time the adapter's event was fired. 

This might result in a change of behavior. If for example an existing basic function block's ECC erroneously triggers the included adapter's event before assigning the variable, the previous value will be now available at the destination as opposed to the new one before.  

It is advised to review and re-test all usages of any basic function block containing an adapter. 

REQ before INIT for SYMLINKMULTIVARSRC

Triggering a SYMLINKMULTIVARSRC function block INIT event is now immediately updating its VALUE data inputs.  

This can cause the loss of pre-existing non-cyclic data inputs values.

Online Change Limitations

Online Change Limitations

The following actions limit the capacity to make online change: 

  • Hardware configuration online changes, except symlinks, require a deploy. 

  • Runtime configuration online changes require a Restart Runtime or a 

Reboot Device to be taken into account. 

The online change might disturb the EtherNet/IP communication and lead to timeout issues, especially when the CPU load is high, that is more than 50 percent. Perform the following steps to prevent timeout issues: #70405 

Step 

Action 

1 

Ensure the setting ENABLE_SYMLIN_OC is set to FALSE in EtherNet/IP bus configuration if you do not need to relearn the symlinks after online change. 

2 

Reduce CPU load by: 

  • Increasing the EtherNet/IP scanner RPI. 

  • Increasing the device communication time-out multiplier (default value is x4). 

3 

Deploy the application to the dPAC. 

Online Change Limitations During Library Development

To perform a successful Online Change of already referenced libraries, follow these recommendations when developing your libraries: 

  • Online Change not compatible with the following block types: 

  • Visible blocks in a library — all block types, CAT, Basic, Composite, SubApp, Adapter and Function 

  • Rename or remove function block types 

  • Rename or remove ports from the interface of function block types 

  • Modifying symbolic link names 

  • Modifying the data type of a port, only if the value can be converted to the new type 

  • Rename or remove UDTs 

  • Rename or remove members of a structure 

  • Reduce the range of a subrange 

  • Reduce an array size 

  • Rename or remove enumeration items 

Online Change is compatible with: 

  • Adding new ports 

  • Adding new symbolic links 

  • Adding new blocks 

  • Algorithm changes and/or ECC changes for basic blocks 

In case blocks or ports of an interface need to be renamed or removed, the current block should be kept unchanged and market as obsolete, and a new block with new interface and name should be added. 

Test Online Change of the modified blocks of the library before releasing the library.
