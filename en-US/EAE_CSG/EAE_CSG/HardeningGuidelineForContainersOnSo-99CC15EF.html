Hardening Containers on Soft dPAC

The purpose of this guideline is to explain the security concerns associated with container technologies and make practical recommendations for addressing those vulnerabilities when specifying, implementing, and maintaining containers. The use of container-specific operating systems (for example, Alpine OS) is more than required to reduce surface attack due to their construction, instead of generic ones (for example, Red Hat, or Ubuntu).

Architectural Point of View

If you modify or add containers, try to group them by functionality or security levels as a measure to containment.

  • Stay minimal: Remove any components that are not needed, or any outdated functionalities.

  • Only use trusted images: Use images that have been previously scanned and found to be trustworthy.

  • Harden the host: Use scripts to configure hosts properly, based on state-of-the-art cybersecurity best practices.

  • Remove privileges: Running privileged containers is a security risk because it creates an opportunity for a malicious user to take control of the host system:

    1. The root access must be limited to designated security member of the company.

    2. The docker members must be limited to designated people when installing and uninstalling the Soft dPAC solution.

  • Consider whether the enterprise, in which the Soft dPAC is deployed, has specific policies on container deployment that need to be applied.
