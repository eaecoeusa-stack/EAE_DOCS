Finalize Generation

The third and final step of the automatic generation of the Modbus configuration is to finalize the import of the Modbus mapping file:

  • Associate Symlinks to the Modbus registers (only if Symlinks haven’t been created during the import).

  • Initiate the Symlinks (only if Symlinks have been created during the import).

  • Set up the last parameters not handled by the automatic generation.

After these steps, the automatic Modbus generation is complete and the Modbus registers is ready for communication.

If the automatic generation didn’t succeed, check the section Troubleshooting for potential solutions.

NOTE: If the issue remains after troubleshooting, consider deleting what has previously been created, then manually develop the Modbus configuration.

Associate Symlinks to the Modbus Registers

If during the import, the symlinks:

  • Have not been automatically associated to the generated Modbus registers: Pre-existing symlinks are associated to them, as detailed in this section.

  • Have been automatically associated: Skip this section and continue at Initiate the Symlinks.

Expand the generated Modbus register. Notice that the value cell of its LinkName line is empty. The associated symlink name will be specified in this cell.

The first step is to determine which symlink to use for a connection (if not already known). Each symlink must:

  • Match with the variable type.

  • Match the usage of the Modbus register:

    • If it is a read Modbus register: The variable comes from a SYMLINKMULTIVARDST function block.

    • If it is a write Modbus register: The variable comes from a SYMLINKMULTIVARSRC function block.

    For ease in identifying a Symbolic Link, its name should have the same symbol on its left as the symbol in the left of LinkName:

NOTE: To identify all the symlinks available for communication in a function block coming from a library, refer to its documentation.

After you have identified a symlinks to associate, add it to the value cell of the LinkName line in one of the following ways:

  • Type the complete name of the symlink inside the single quote marks.

    The typed name must contain the name of the variable transmitting the value preceded by all the names of the function blocks in which the variable is imbedded. Each function block name is separated from the next function block name by a dot.

    Example: 'Valve.output.cmdActive.IOSignal'.

    NOTE: Save All after the name is added to refresh the symbolic links list. If an input error is detected, the Modbus register will appear at the top of the list with an exclamation mark indicating the association didn’t succeed.

  • Drag and Drop the symlink from the Symbolic Links tab into the cell to make the association. The complete name of the symlink is automatically added into the cell. Adding a symlink this way helps prevent detected input errors.

Perform this association action for all Modbus registers added during the automatic generation.

Configure the Symlinks initialization

If during the import, the symlinks

  • Have been automatically associated to the generated Modbus registers: Their initialization values are configured.

  • Have not been automatically associated: Skip this section.

As mentioned in the Import Modbus mapping File topic, during the automatic generation of the Modbus registers, symlink function blocks are added to the resource network of the system using the Modbus communication.

At the conclusion of the process, each Modbus register has its symlinks created in the resource network. These symlinks are added to new SYMLINKMULTIVARDST and SYMLINKMULTIVARSRC function blocks.

The newly added function blocks are connected together, their INIT and INITO ports are linked together to propagate the initialization event.

The only remaining step is to link the INIT port to get the initialization trigger and to initialize all the symlink function block. To do this, link the INIT port to the INITO of an already existing function block in the resource or to the COLD port of the E_RESTART function block.

NOTE: It is possible that during the generation, all the created symlink function blocks are located at the same position. If this occurs, reorganize them into the resource layer.

Set Up the Last Parameters

To complete the automatic generation and have the Modbus communication correctly set, check the value of few additional parameters.

These parameters are not handled by the automatic generation because their default values are generally sufficient. These variables are:

  • Swap; available by expanding the Modbus register, it defines the reading rule of the double and long registers. It is set to FALSE by default. When set to TRUE, the reading order of the bytes composing the register are inverted.

    NOTE: This parameter is generally set to TRUE when the parameter MSB also has been set to TRUE.

  • ioevent; available in the Properties of the Modbus register, it defines how the information is communicated:

    • cyclic; selected by default, this setting sends the value on a cyclic basis, even if the value has not changed.

    • requestwrite; available only for write Modbus registers, it sends the value on a trigger event basis, only when the value has changed.

      NOTE: It is recommended to choose requestwrite for writing Modbus registers. The only reason to choose cyclic for writing Modbus registers is to help prevent another device from overwriting the value.

    • requestread; available only for read Modbus registers, it enables the Modbus register to read IO value on event base instead of cyclically.

  • modbusfunction; available in the Properties of the Modbus register, it defines how the Modbus server should read the value sent by the writing Modbus client:

    • (Default); when this setting is chosen, it works the same as Write Multiple Registers (FC16).The server will read as a whole the bytes of the Modbus register.

    • Write Single Registers (FC06); this setting causes the server to read individually each byte of the Modbus register. This setting induces many more requests to the Modbus server. Consider using this setting only if the device reading the Modbus client requests does not support the Write Multiple Registers (FC16) setting.

    NOTE: This setting is only available for the writing Modbus registers transferring register.

  • reqReadLinkName; available only for read Modbus registers, configure it only when ioevent of the Modbus register is set to requestread. Give to this parameter a symlink name from a SYMLINKMULTIVARSRC function block. This symlink function block has to:

    • Set as INT its value type.

    • Set its value to 1 and set its QI input to TRUE.

    • Once online, trigger its REQ input event every time you want to read the Modbus register value.

After completed setting the value of these parameters, the Modbus communication is ready to be tested and used.

Troubleshooting

To find out if the Modbus communication is usable, a few actions can be performed, for example, compiling the application, testing communication with a Modbus server simulation application and so forth.

After running these tests, check the parameters of the Modbus registers failing a test.

The following are known events you may encounter during the generation. A potential response is suggested for each one.

Error in the XML document

If during the import of the Modbus mapping file the detected error message ‘There is an error in the XML document’ is displayed, it means that the Modbus mapping file does not conform to the Modbus Gateway standard. Some modifications may have been done in the XML. Consider re-creating the Modbus mapping file from a freshly re-unzipped Excel file.

Does not contain any item

If during the import of the Modbus mapping file the detected error message ‘[The Modbus mapping file] Does not contain any item’ is displayed, it means that the Modbus mapping file is empty. Re-export the mapping file, taking care to include all the content inside the Excel table.

Modbus register doesn’t appear

After the import is complete, if the Modbus register doesn’t appear below the Modbus device, it could mean that inconsistent values have been given to some parameters in the table. Some inconsistence can be:

  • Misspelling word in the Type, Usage, isIO or BigEndian column.

  • Including a number outside of the accepted array of values for the Offset column.

    NOTE: With this inconsistence, it may still generate the Modbus register, but the address parameter in the Properties will be empty.

  • Assigning incorrect parameters for the Type of the Modbus register. For example, including a value in a column that should remain empty.

Check the Excel table and correct the potential issues according to the rules detailed in the topic Create Modbus mapping file.

Despite the errors in the Modbus mapping file being corrected, it is possible that the Modbus registers may still not appear in subsequent imports. In this case, delete the Modbus device where the issue occurred, then add a new device and repeat the import to it.

Modbus register replaced

When an import is done into a Modbus device that already contains Modbus registers, it may occur that the new Modbus registers have replaced the pre-existing ones, instead of being added at the end of the pre-existing registers. Consider avoiding adding new content to a Modbus device already containing some registers.

To add new content either:

  • Add a new Modbus device where the new Modbus register will be added (if the new Modbus registers can be communicated separately from the old ones).

  • Create a Modbus mapping file containing both the old and the new Modbus register and import it in a new Modbus device, and delete the old device (if the Modbus registers have to be communicated to the same server).

  • Add manually the new Modbus register into the Modbus device by following the manual development of the Modbus registers steps.

This list of issues is not exhaustive. If some issues remain even after troubleshooting, it is recommended to switch to the manual development of the Modbus register.
