Elements

The elements of a project are: Service function blocks, Basic function blocks, Composite function blocks, SubApp function blocks, Devices, Resources, Applications, System, CAT and SCADA (includes canvas and graphics for monitoring, control and data collection).

These elements, with the exception of Service function blocks, can be copied, deleted, renamed as well as exported or have their references displayed by right-click on them. Then select the corresponding item from the sub menu.

Service function blocks can only be opened and have their references displayed. Service function blocks are provided within ready-to-use libraries. Some of them are designed as generic function blocks, that means the number of in- and/or outputs can be adjusted in accordance with the needs of your application. See also generic function blocks.

Context Menu

Right-clicking an element in the Solution Explorer opens a context menu with the following commands:

Command

Description

New Item...

Opens a dialog to create a new object, depending on the selected node (Basic, Adapter, CAT, ...).

New Folder

Creates a new folder in the selected node.

Rename

Opens a dialog to rename the selected element.

Remove

Deletes the selected element.

Paste

Pasting the content of the buffer memory.

Exports the content of the selected node (all elements of the node, when a node is selected, or the single element, when only one element is selected). A dialog opens to define the export folder.

Opens a dialog to import the desired elements (like *.Basic.zip, *.cat.zip, ...).

Fieldbus Components...

Opens the Fieldbus Components editor.

Opens context-sensitive help. (Also available on pressing F1 key.)

When to use a Basic and when to use a Service Function Block?

The Basic Function Block represents a function block with passive functionality, that means it has to be triggered via one of his input events to force executing its logic and firing its own output event. It is depicted only by its algorithm execution logic and it is used when its functionality can be expressed using an algorithm.

The Service Function Block may also be an active function block, that means it can also produce output events without receiving an input event (e.g. E_CYCLE). Additionally, it can provide operating system features (such as file access, communication, transfer, and such).

When to use a Composite and when to use a CAT Function Block?

A Composite and a CAT are basically the same, they represent a network of different function blocks and they can also be active (as can ServiceFBs), because they may contain Service FBs as well as Basic FBs or even complete Composites. A CAT behaves similarly to a Composite, but a CAT function block additionally contains an HMI-Interface-function block which represents an interface for value exchange with the HMI.

Within this HMI function block the HMI symbols or faceplates can be created. These can be instanced in the respective Canvas, according to usage and requirements, and serve as a direct exchange platform between HMI and project values. With this HMI-Interface function block the whole HMI-binding of the project is configured automatically. (Only the HMI-Device also has to be configured.)

Elements - information

NOTE:

  • The size of Basic-, Composite-FBs and CATs can reach a value up to 51200 Bytes. If an error of this kind is detected during deployment, distribute the functionality of this function block among two or more FBs.

  • Since all items defined in the configuration result in HWCAT inputVars, and one HWCAT has a limit of 256 inputVars, EIPSCANNER cannot handle configuration where number of items exceeds this limit. Therefore user needs to manually migrate to EIPSCANNER2.

    An HMI Function block can reach a number up to 30 input and 30 output events, whereas the INIT-Events as well as input data events associated to persistent variable (check out Offline Parameterization Editor topic) are not to be counted. For data ports of HMI FBs also the number of 254 ports applies, as described above.
