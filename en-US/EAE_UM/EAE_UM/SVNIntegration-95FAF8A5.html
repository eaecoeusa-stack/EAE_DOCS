SVN Integration

NOTE: This functionality might become obsolete in a few EcoStruxure Automation Expert version.

The Apache® Subversion® system (SVN) is used to create, configure and manage revisions of your projects over time and to facilitate the collaboration of several designers whose modifications can be unified.

The components used to provide SVN support are installed when installing EcoStruxure Automation Expert - Buildtime. 

Additionally it is possible to install other clients when SVN is also needed to manage other developments, which are independent from EcoStruxure Automation Expert - Buildtime.

It is recommended to manage EcoStruxure Automation Expert - Buildtime projects exclusively with the integrated SVN functions. Client version 1.10 (or higher) has been equipped with an additional merge tool for XML-based projects to facilitate the simultaneous editing of a project by several users. You can find the setting Use XML-based merge tool in the dialog Settings at item Subversion (SVN) .

NOTE: When creating an SVN archive (by other means than the EcoStruxure Automation Expert - Buildtime integrated tools) use only one of the following SVN versions: 1.7, 1.8, 1.9, 1.10.

Creating an SVN Repository

To create an SVN repository: 

  1. Right click the Solution (or project, or project library) node in the Solution Explorer tree view.

  2. Click Create Subversion Repository.

The following sub-context menu items appear:

Command

Description

Create Subversion Repository

Opens a dialog to select or create a folder (repository) to store the SVN archives. If a folder is selected that is already used as an SVN repository or which includes any other content, a detected error notification is displayed. In that case, select a different, empty folder or create a new one.

NOTE:

  1. The path of the new local repository is copied to the clipboard and can be used directly for the Add Solution to Subversion dialog.

  2. When the repository is created using thisEcoStruxure Automation Expert - Buildtime-integrated function, it is automatically compatible with each of the above mentioned SVN client versions.
Add Solution to Subversion

Opens the Select SVN Repository dialog, which you use to add the currently opened solution to an existing SVN archive. The modifications of this solution can then be managed by this repository. Copy the path of the respective repository to the Repository text field in the dialog, and the current project name will be added automatically.

When adding the solution to a SVN repository, a dialog box may appear to inform about the incompatibility of SVN and Online Change. Every update in the application needs a full deployment. Confirm this dialog box to open the Select SVN Repository dialog.

This dialog box appears if:

  • The solution is added for the first time to a SVN repository.

  • The solution is logged in to a device.

  • The application is deployed into the device.

Configuring Repository URLs

You can access subversion repositories using several different methods:

  • on local disk

  • via several network protocols

Every repository location is addressed as a URL. The URL schema determines the access method:

Schema

Access Method

file://

Direct repository access on local or network drive.

http://

Access via WebDAV protocol to Subversion-aware Apache server.

https://

Same as http://, but with SSL encryption.

svn://

Unauthenticated TCP/IP access via custom protocol to a svnserve server.

svn+ssh://

Authenticated, encrypted TCP/IP access via custom protocol to a svnserve server.

As a general rule, subversion URLs use the standard syntax, which allows server names and port numbers to be specified as part of the URL. The file:// access syntax is typically used for local access; however, it can also be used with UNC paths to a networked host. In this case, the URL would take the form file://hostname/path/to/repos. If the target is the local machine, the hostname portion of the URL is either absent or localhost. For this reason, local paths normally appear with three slashes, file:///path/to/repos.

In addition, users of the file:// scheme on WindowsTM platforms need to use the  unofficial standard syntax for accessing repositories located on the same machine, but on a different drive than the current working drive. Either of the following two URL path syntaxes will work where, for example, the repository resides on the X drive:

file:///X:/path/to/repos

...

file:///X|/path/to/repos

...

NOTE: : A URL uses forward slashes despite the fact that the native (non-URL) form of a path on Windows uses backslashes

Managing Versions of Subversion

The program version of Subversion can be chosen in Settings at item Subversion (SVN). If the chosen version of Subversion is not identical with the installed version of Subversion, a detected error message appears as soon as you want to open a solution that is under control of Subversion.

These possibilities are offered there:

  1. Upgrade Subversion to that version chosen in EcoStruxure Automation Expert - Buildtime,

  2. Use the same version of Subversion as installed on machine, or

  3. Select Ignore and do not use Subversion at all.

Modifications of individual files in a version controlled project are managed in EcoStruxure Automation Expert - Buildtime in a project-specific History tab presenting its Revision history. Select an item in the Revision list (in the top of the tab) to display the project modification details – which have been managed via SVN – in the Action / Path window area (in the bottom of the tab).

NOTE: This sub-tab is available only after the first SVN commit action.

Any available SVN actions can be triggered via the context menu in the Solution Explorer if the opened project is an SVN-versioned project. Refer to the description, located in the lower part of the topic, for the Distributed_PAC_Project context menu.

As soon as an SVN action, such as Commit, Update, and so forth is started, any open editors are automatically closed temporarily and reopened after the process is finished. After the process has been finished, the respective editors will again display the updated content of the corresponding project file.

If you want to open a project that is still in a conflicted state, the Information dialog opens to indicate the solution has unresolved conflicts and asks you what action you want to take:

 CAUTION
RISK OF DAMAGE TO PROJECT
Selecting the Open despite conflicts option may cause damage to your project. Opening your project using this commend is not recommended.
Failure to follow these instructions can result in injury or equipment damage.

Available actions include:

Command

Description

Open despite conflicts

Opens the project despite existing conflicts.

NOTE: this option is not recommended because it may damage the project.
Cancel

Closes the dialog and cancels the opening of the Solution.

Resolve

Closes the dialog and opens the SVN dialog showing the list of all conflicted files. To solve the conflicts open the context menu by right-clicking on one of the conflicts in the SVN dialog and select the entry Edit conflicts, as described in the topic Merge Tool.
