Log Files

Log files contain protocol information about the project and its devices. They can be used to help you identify specific detected errors within the project. For each device, two log files are stored which are overwritten alternately, so that data will not be lost. In addition, a separate file is maintained that records cyclically recurring detected errors which you can use to quickly identify these events.

Adjusting the Log File Size

The log file size is set to 65 kByte by default. You can adjust it at item Debug > Log > File > MaxFileSize for the selected device properties in the right part of the Logical Devices editor.

NOTE: Twice the log file size set here will be used on the device, because there are two log files stored on each device.

Only the first occurrence of a cyclical detected error is recorded in the cyclic log file. However, if the same error is detected multiple times, a Count is maintained indicating the number of times that cyclic error has been detected. To open the Log files window, select View > General pads > Deploy and Diagnostic, then in that window select Action > Log file > Fetch Log File. To display a description of the detected error, select a cyclic log item in the top pane, then click Details to display its ID, count, and a description.

The value of the Count attribute is reset to 0 when:

  • The Action > Deploy > Clean command is executed for a device.

  • The device stops, as a result of a detected error.

  • The function block, or other source cause of the event, is removed from the program.

The file for cyclic detected errors will not be stored on the device but it is only held in working memory and available only online.

The maximum number of allowable cyclic log file entries can be configured in the Deploy and Diagnostic editor in the selected device Properties window at Runtime Configuration > Debug > Log > Cyclic > MaxEntries.

You can use the Log File window as a source of information when working with Schneider Electric technical support to resolve a detected error. To do this:

Step

Action

1

Open the Log File window by selecting View > General pads > Deploy and Diagnostic, then in that window select Action > Log file > Fetch Log Files.

2

In the top pane, select the specific event (for example, a detected error) that, you need to resolve. The details of the selected item are displayed in the Message Details (lower) pane.

3

Click Copy to copy the message details into the PC clipboard.

4

Paste the details from your PC clipboard to a file that you can save and send to Schneider Electric technical support.

NOTE: In case a Log message like: Too many events in an event chain, THROTTLING for 10 ms appears, the settings of the relevant device have to be adjusted. Open the Logical Devices editor and navigate within the properties of the device to: Runtime Configuration > Resource > EventQueue > Throttle. Here you can adjust the threshold values LimitTime, PauseTime, HaltNumber accordingly. This will set the threshold values for the number of events being processed without an idle phase of the controller. A description of the particular value is shown in the Logical Devices editor beneath the properties, when the relevant value is selected.

After the device settings have been modified, the new configuration has to be deployed to the device. See Deploy and Diagnostic editor.

HaltNumber will also cause an ErrorHalt in the case that an event loop exists in the project.

NOTE: The current state of the device is shown in the Logical Devices editor. If the state ErrorHalt occurs the event processing will be halted, because there had been too many events, so that there is a high probability for the occurrence of an endless-event-loop in the project. In that case, please check the log (See: Deploy and Diagnostic editor). You can see the current content of the event chain there, which may give you a hint where to find the loop by checking which function block has fired which event.

Handling a Log Buffer Overflow

For advanced users: In some cases, if the event-chain is extremely long, the log buffer could get an overflow. So if you have an incomplete event-chain after an incident like this, the config-entry Configuration > Debug > Log > RingBufSize could be increased (max. 64k)

Example of a log:

2012-12-17-17:56:47.916 E: SrcID: 0 SubID: 16: Dump of event chain: Start

2012-12-17-17:56:47.916 E: SrcID: 0 SubID: 16: INTERN: 0.000091130: RES1.FB5.DEVLOG.DEVLOG.1

2012-12-17-17:56:47.916 E: SrcID: 0 SubID: 16: INTERN: 0.000091130: RES1.FB5.DEVLOG.DEVSTATS.1

2012-12-17-17:56:47.916 E: SrcID: 0 SubID: 16: Dump of event chain: End

If a device remains in state Deploying without going on, just clean it and deploy anew.

NOTE: The function block LOGGER, from the Runtime.Base library, can be used to write user defined messages into the log. The logging level of the device can be defined in accordance with the requirements. If the logging level of the device is set to Debug, also Debug messages are available in the log file. The default setting is Info, that means only info, warning and detected error would be available, but no debug messages. This setting avoids an overstressing of smaller devices. In order to write debug messages to the log, open the Logical Devices editor, select the device and open Configuration > Debug > Log > Level in the device properties. Set Level to Debug (or e.g. Warning > no Info and no Debug messages are written to the log file, etc.).
