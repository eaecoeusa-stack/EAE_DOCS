1. Scope

The following rules apply to VTQ data types as implemented in EcoStruxure Automation Expert Buildtime software:

Basic VTQ data types:

  • Standardized, interoperable VTQ data types are system data types that correspond to a limited set of elementary IEC 61131 data types.

String data types and VTQ:

  • VTQ supports fixed size STRING types, but does not support dynamically sized DYNSTRING types.

User Defined Types (UDTs) and VTQ:

  • VTQ data types do not support user defined types (UDTs); for example, "VTQUDT", "VTQMyStruct", "VTQMyArray", and so forth, will not exist.

  • VTQ data types cannot be incorporated as elements within a UDT.

Enumerated data types and VTQ:

  • VTQ data types do not support Enum types.

  • VTQ data types are Class 1 types with definitions fixed at runtime, while Enum data types are Class 2 types whose definitions are not determinable at compile time.

VTQ support for Standard function block Connections:

  • Standard IEC 61499 function block connections support all new VTQ types.

Implicit type casting and VTQ:

  • Implicit type casting, without the need for an ANY2ANY function block, between VTQ types follows the same rules as implicit type casting between elementary types.

  • When implicit type casting from an elementary type to a VTQ type, all VTQ elements except Value are set to default values:

    • Q is set to 0x00960000, defined as “UA_SCGOODLOCALOVERRIDE” by OPC

    • T is set to 0.

  • When implicit type casting from a VTQ type to an elementary type, only the Value is used. T and Q values are stripped off.

Explicit type conversions and VTQ:

  • These conversions can be handled by Encoder/Decoder function blocks or the ANY2ANY function block.

  • For the Encoder function block (VTQ_ENCODE), the user controls T and Q values when converting from non-VTQ to VTQ.

  • T and Q values are stripped off when converting from VTQ to non-VTQ.

  • The ANY2ANY function blocks use the same convention as for implicit type conversion in terms of setting T and Q values. 

Generic function blocks (Dynamically Generated Function Block data interface) and VTQ:

  • Generic function blocks support both VTQ and other (existing) data types:

    • SYMLINKMULTIVARSRC and SYMLINKMULTIVARDST, VTQ_ENCODE/VTQ_DECODE and ANY2ANY (for explicit casting) are supported

  • Connections from non-VTQ to/from VTQ data types on Generic function blocks follow the same type casting rules as other non-VTQ to VTQ.

  • Implicit elementary to VTQ cast with user-defined initial TQ values uses the initial declared Value; Q and T set to 0.

Symbolic links and VTQ:

Data types for Symbolic Links are determined by the SYMLINK function block parameter(s).

NOTE:

  • Symbolic links support all VTQ types.

  • Symbolic links are generic (dynamic) function blocks that support automatic type conversion between VTQ and non-VTQ types, including:

    • VTQ-to-VTQ and non-VTQ-to-non-VTQ connections

    • VTQ-to-non-VTQ and non-VTQ-to-VTQ as long as the basic types of the “value” fields are the same

    • For conversion, the basic types are supported; for example from VTQREAL to REAL or REAL to VTQREAL.

    • T and Q follow the same rules as for implicit type conversion.

HMI and SCADA functions (Historians, etc.) and VTQ:

  • HMI support all VTQ data types.

  • The persistence of the VTQ variable is possible as an optional parameter.

  • VTQ Variables can be archived, but only the value part. For more information about archiving a variable, refer to the EcoStruxure Automation Expert - Archive User Guide, available in the Archive subfolder of the EcoStruxure Automation Expert installation package.

Structured Text Applications and VTQ:

  • IEC 61499 Structured Text (ST) algorithms support VTQ by means of “dot-notation”. No implicit type casting between VTQ and non-VTQ types is allowed. (For example, the value (V) of a variable “Input1” of type VTQREAL is accessed using the syntax Input1.value).

  • There is no implicit type conversion for ST algorithm. Instead, dot notation is used. For example, ADD(varVTQINT.value, varVTQINT.value).

  • ST Functions that support UDTs also support VTQ types. An example is the SELECT() function where all arguments as well as output are same type. Other examples include: MOVE, DEMUX, EQ, NEQ.

Existing function blocks and VTQ:

There is no immediate impact on existing IEC 61499 function blocks. The following minimal changes are made to existing function blocks:

  • VTQ types can be added to, but initially are not supported in existing EcoStruxure Automation Expert libraries.

  • Separate libraries with full VTQ support can be added.

Watch, Constants and VTQ:

  • VTQ is supported in Watch, Constants and Initial value.

Force and VTQ:

  • VTQ is supported as a forced value so that function block inputs/outputs can be overridden for a function block instance in a project.

Online Change and VTQ:

  • Online Change supports VTQ (Add/remove/modify variable/constants/initial value).

  • An existing VTQ variable changing to another type will retain its value (V), as this depends on implementation of ST VTQ-ELELMENTARY cast functions.NOTE: This does NOT apply to Generic function blocks because a change in input/output data connections or data type(s) results in a new block type, and all data is re-initialized.

  • The “value” of a VTQ variable will be converted (cast) just like any other IEC 61131 type For example changing max size of VTQSTRING.

Licensing and VTQ:

  • VTQ support is enabled and disabled as a licensed feature in Buildtime. If disabled:

    • VTQ types will not be available when defining Function Block interfaces.

    • Compiler errors will be generated for any function block that has a VTQ type on its interface.
