1. ERR Error Code List

ERR_ADAPTER_USES_DYNSTRINGS

Adapter A, called in the basic B, uses dynamically allocated string

The basics function blocks, unlike the other types of function blocks, only support statically allocated strings with a fixed maximum number of characters. Thus, two cases scenario are possible:

  • A string is defined without a fixed maximum number of characters into a basic: A default size is assigned to it and the warning WRN_UNSIZED_STRING is shown to inform the user.

  • A dynamically allocated string is transferred into the basic through an adapter: The basic cannot support the dynamically allocated string and this error is shown.

Here, the adapter A uses a dynamically allocated string and is called in the basic B. Thus, it corresponds to the second case scenario from the list above.

Example Case 

ADAPTER Adpt
	EVENT_INPUT
		E_AREQ;

	END_EVENT
	EVEN_OUTPUT
		E_ACNF;

	END_EVENT
	VAR_INPUT
		aIn : STRING;

	END_VAR
END_ADAPTER
	
FUNCTION_BLOCK ERR_ADAPTER_USES_DYNSTRINGS
	EVENT_INPUT
		E_REQ;

	END_EVENT
	EVENT_OUTPUT
		E_CNF;

	END_EVENT
	VAR_OUTPUT
		out1 : STRING;

	END_VAR
	SOCKETS
		sck1 : Adpt

	END_SOCKETS
	EC_STATES
		START;
		ST_REQ : A_REQ -> E_CNF;

	END_STATES
	EC_TRANSITIONS
		START TO ST_REQ := E_REQ;
		ST_REQ TO START := 1;

	END_TRANSITIONS
	ALGORITHM A_REQ IN ST :
		out1 := sck1.aIn;

	END_ALGORITHM
END_FUNCTION_BLOCK

This example generates the following error and warning while compiling: 

In VarDeclaration 'out1': Type 'STRING' is not explicitly sized, using a size of '15' by default (WRN_UNSIZED_STRING)
In AdapterDeclaration 'sck1': Adapter 'Adpt' uses dynamically allocated string (ERR_ADAPTER_USES_DYNSTRINGS)

This example is then first creating the adapter Adpt with the dynamically allocated string aIn as no fixed size is mentioned.

Then the basic ERR_ADAPTER_USES_DYNSTRINGS is created. It has two states, START and ST_REQ. When the input event E_REQ is triggered, it moves to the state ST_REQ and executes the algorithm A_REQ. Once done, it triggers the output event E_CNF and moves back to the START state.

The algorithm A_REQ gives to the output variable out1 the sck1.aIn value (sck1 being an Adpt adapter).

  • The warning message corresponds to the first case scenario from the list above: The output variable out1 size is not specified during its definition.

  • The error message corresponds to the second case scenario from the list above: The adapter Adpt, used as type by the socket sck1, uses the dynamically allocated string aIn.

TIPS

Right click in the error and select Show in Editor, this will show where the error is located. Then either:

  • Open the adapter used in the basic and specify a fixed maximum number of characters for the string. In the example case, it would be: 

    VAR_INPUT
		aIn : STRING[15];

END_VAR
    NOTE: The size 15 given here to aIn is an example. It doesn’t necessarily need to be 15.

  • Use another type of function block, like composite, and redo the algorithm inside it. As all other function blocks than basics handle dynamically allocated string, then the issue won’t appear by using another function block type.

ERR_ALGPARAMETER

Nonstandard parameter defined in an algorithm

The current standard is not supporting the input or the output of variable in the Algorithm editor of a Basic function block.

Thus, the following variables cannot be defined in a Basic algorithm:

  • VAR_INPUT

  • VAR_OUTPUT

  • VAR_IN_OUT

Because, in the Execution Control Chart editor, it is not available to define input or output variables when selecting the algorithm of a state.

Example Case 

FUNCTION_BLOCK ERR_ALGPARAMETER
	EVENT_INPUT
		E_REQ;
		
	END_EVENT
	EVENT_OUTPUT
		E_CNF;
		
	END_EVENT
	VAR_INPUT
		QI : BOOL;
		
	END_VAR
	VAR_OUTPUT
		QO : BOOL;
		
	END_VAR
	EC_STATES
		START;
		ST_REQ : A_REQ -> E_CNF;
		
	END_STATES
	EC_TRANSITION
		START TO ST_REQ := E_REQ;
		ST_REQ TO START := 1;
		
	END_TRANSITIONS
	ALGORITHM A_REQ IN ST :
		VAR_INPUT
			A_REQ_VAR : BOOL
			
		END_VAR
		
		QO := QI AND A_REQ_VAR;
		
	END_ALGORITHM
END_FUNCTION_BLOCK

This example generates the following error while compiling: 

In Algorithm 'A_REQ': Nonstandard parameter list for algorithms (ERR_ALGPARAMETER)

In this example, the function block receives QI as an input variable and attempts to receive a second input variable A_REQ_VAR directly from the algorithm. The result of the logic AND between those two BOOL variables is transmitted to QO.

The error appears because it is not standard to define input or output parameters directly inside the algorithm of the function block.

TIPS

Right click in the error and select Show in Editor, this will show where the error is located. Then:

  • Define the variable issuing inside the Interface editor of the function block instead of inside the algorithm.

ERR_ATTR_OPTARG

The argument in the compiler option is incorrect

When evaluating pragmas, an invalid argument has been added in the compiler option.

Compiler option pragmas are used to enable or disable compiler features which are normally passed to the compiler on the command line.

Tips

Right click in the error and select Show in Editor, this will show where is located the invalid argument. Then modify the compiler pragma line so it follows this structure:

{ CompilerOptions := '/1:2' }

With:

  1. The argument: Either enable or disable.

  2. The feature: Add the name of the feature from those defined in enumeration IECFeatureList::FeatureID. All features may be used but the ones marked 'internal' (those which are not displayed in the help text of the type compiler) should be used with awareness of its possible misuse. An error is detected if the argument list does not start with an enable or disable keyword or if the feature argument is not one in the feature list.

ERR_BOOL_SHIFT

Shift and rotate operations on operands of type BOOL are not allowed

As the type BOOL is made of only one bit, doing a shift or rotate operation on it is unnecessary.

This error appears when a BOOL variable is given as an input to the following functions:

  • SHL: Shift the operand value to the left.

  • SHR: Shift the operand value to the right.

  • ROL: Rotate the operand value to the left.

  • ROR: Rotate the operand value to the right.

Example Case 

FUNCTION BOOL_SHIFT
	VAR_INPUT
		I1 : BOOL;

	END_VAR
	VAR_OUTPUT
		O1 : BOOL;

	END_VAR

	O1 := SHR(I1, 1)

END_FUNCTION

This example generates the following error while compiling: 

In Algorithm 'BOOL_SHIFT': Shift and rotate operations on operands of type BOOL are not allowed (ERR_BOOL_SHIFT)

This example attempt to shift to the right by 1 digit the input I1. Once done, the function would return the result of the shift to the output O1.

The error appears as the input I1 is from the type BOOL.

TIPS

Right click in the error and select Show in Editor, this will show where the error is located. Then either:

  • If the BOOL type of the input is cannot be changed: Remove the shift or rotate function creating this error as this function is not necessary.

  • If the shift or rotate function cannot be removed: Change type of the input and output of the function so the shift or rotate can be done.

ERR_CAST_CONSTANT

The constant conversion of type [Constant type] into [Variable type] is not allowed or undefined

When declaring a variable, the type of the constant value needs to be either the same as the type of the variable declared, or convertible. Thus, this error detects that the constant type cannot be converted into the variable type in the variable declaration.

This rule also apply for all single elements in arrays and all structure elements of structures.

Example Case 1 

FUNCTION ERR_CAST_CONSTANT
	VAR
		in1 : USINT := 'X';

	END_VAR
	
END_FUNCTION

Example Case 2 

FUNCTION ERR_CAST_CONSTANT : VTQINT
	VAR
		in1 : VTQINT := 12345;

	END_VAR

	ERR_CAST_CONST := in1;

END_FUNCTION

These two examples generate the following errors while compiling: 

In Algorithm 'ERR_CAST_CONSTANT': Cannot convert constant of type 'CHAR' to type 'USINT' (ERR_CAST_CONSTANT)
In Algorithm 'ERR_CAST_CONSTANT': Cannot convert constant of type 'ANY_INT' to type 'VTQINT' (ERR_CAST_CONSTANT)

During the declaration of the first function, a constant of type CHAR is assigned to the variable in1 of type USINT. The error then appears as the conversion requested to the variable in1 is undefined.

During the declaration of the second function, a INT value is assigned to the variable in1 of type VTQINT. The error then appears as the VTQ type of the value assigned is not defined.

TIPS

Right click in the error and select Show in Editor, this will show where the error is located. Then either:

  • Check that the constant type is matching with the variable type. Like in example case 1, either the type of in1 has been wrongly chosen or the constant has been wrongly assigned.

  • Check that nothing is missing in the assignment. Like in the example case 2, the VTQ type hasn’t been defined. In that case, one option is to assign the constant as followed: in1 : VTQINT := (value := 12345);

  • If the two constant and variable types are correct and the conversion is required, perform the conversion in the development area with a converting function, not in the declaration area.

ERR_CG_EXPRESSION

${1} (${2},${3}) ${eCls} ${eSym}: Expression too complex

The expression in the given source code line is too complex and the backend runs out of registers for intermediate result evaluation. Currently only floating- point expressions on an Intel x86 processor are known to have this limitation. To fix this detected error the expression can be split up into two or more simpler expressions which have their results stored in temporary variables.

ERR_CG_FCTNOTFOUND

${1} (${2},${3}) ${eCls} ${eSym}: Function '${4}' not found

The function named in parameter ${4} was not found in the current runtime library. The named function may either be an internal or a user defined one. In case of an internal function the program usually tries to use a functionality, the runtime is not configured for (e.g. trying to take the sine of a LREAL number on a runtime system which does not support long real numbers). In case of a missing user defined function the download process is possibly broken. You may try to recompile the project which may detect missing references or fix broken dependencies.

ERR_CG_ICODE

${1} (${2},${3}) ${eCls} ${eSym}: Internal intermediate detected error #${6} '${4}:${5}'

This is also a compiler internal non-recoverable detected error which is given if an intermediate code instruction contains an undefined operand combination or invalid argument values. Parameter ${4} will be assigned the name of the library element being currently processed, parameter ${5} the number (address) of the non-performing intermediate code instruction and parameter ${6} an unique identifier for the operation which did not succeed. Submit detected error message and if possible also the compiled library element containing the non-performing code (the .bin file) to your software supplier.

ERR_CG_LIBLOAD

${1} (${2},${3}) ${eCls} ${eSym}: Did not successfully load external library '${4}'

This detected error is emitted by the test container if it does not successfully load a user supplied C/C++ library implemented as DLL or shared object. This feature is currently in an experimental state, tested for x86 platforms running Win32 only and therefore not available to the public.

ERR_CG_NOTSUPPORTED

${1} (${2},${3}) ${eCls} ${eSym}: Code generation: '${4}' is not supported

The functionality given in parameter ${4} is not supported by the current restricted back end. Missing features correspond with configuration macros RTS_SUPP_.... Possible values for parameter ${4} are: '64-bit integer' (RTS_SUPP_I64), 64-bit real numbers (RTS_SUPP_R64), 32-bit real numbers (RTS_SUPP_R32),.... To resolve the detected error, verify that the described feature is not used in ST source code.

ERR_CG_RELOCATION

${1} (${2},${3}) ${eCls} ${eSym}: Relocation detected error at offset ${4}

This detected error is generally given when a native code instruction is emitted were some fields are still unknown. When the value for these fields are known later during code generation, the value is found to be too large to fit into the instruction. The detected error is non-recoverable and cannot be fixed easily. Some fiddling with the code may work but in general is advisable to report a compiler internal detected error.

ERR_CG_TOOLKIT

${1} (${2},${3}) ${eCls} ${eSym}: Internal detected error '${4}'

This detected error reports internal non-performance in the code emitter toolkit. Parameter ${4} names the reason for the non-performance. Report a compiler internal detected error.

ERR_CHARSET

(${1},${2}) ${eCls} ${eSym}: Identifier '${3}' uses extended character set

Basically, the compiler accepts programs encoded in UTF-8-character set implying that identifiers can also include characters from foreign character sets (Greek, Russian, Hebrew, Arabic, ... to name a few). This is an extension to  the standard which allows plain ASCII characters only.
NOTE: For the compiler to accept foreign character sets compiler feature ExtCharSet must have been specified on the command line, otherwise ERR_CHARSET is given.

ERR_COMMENT

(${1},${2}) ${eCls} ${eSym}: Scanner detected error: unterminated comment

A multi-line comment has no corresponding end symbol. Nesting multi line comments does not work in this compiler version.

ERR_CONF_CROSS_COMM

Incorrect configuration of Reliable Cross Communication

EcoStruxure Automation Expert has been configured in a way that prohibits Reliable Cross Communication. Thus, the communication of events or variables between the function blocks FbA and FbB can not be done as these two function blocks are mapped in different devices.

Example Case 1

The solution contains:

  • The library SE.DPAC version lower than 23.0.0.10

  • A function block FbA and a function FbB.

  • Function blocks FbA and FbB not mapped on the same device.

  • An event and a variable directly linked between the function blocks FbA and FbB.

    NOTE: The variable linked is associated with the event linked.

Example Case 2

The solution contains:

  • The library SE.DPAC version equal or greater than 23.0.0.10

  • A function block FbA and a function FbB.

  • Function blocks FbA and FbB not mapped on the same device.

  • An event and a variable directly linked between the function blocks FbA and FbB.

    NOTE: The variable linked is associated with the event linked.

  • The parameter Runtime Configuration > Communication > Reliable CrossComm > Protocol > Disable Reliable QoS all devices set to TRUE.

These examples both generate the following error while compiling: 

The EcoRT_1 device does not support the "Resend a message if it is not acknowledged" QoS level, please use the "Send and Forget" QoS level for the event "StopO" (double click on this message to go to the relevant FB) or unset the device property "DisableReliableQoS" (uses more CPU and RAM) (ERR_CONF_CROSS_COMM).

These examples contain two function blocks FbA and FbB communicating through cross communication as they are mapped on different devices.

In both examples the cross communication is failing. It is due to:

  • In example case 1: The library used is older than 23.0.0.10 and does not support Reliable Cross Communication.

  • In example case 2: The parameter Disable Reliable QoS of at least one of the devices is set to TRUE.

TIPS

Right click in the error and select Show in Editor, this will show where the error is located. Then either:

  • Map one of the two function blocks into the other device to avoid cross communication.

  • If the cross communication is not avoidable:

    • Verify the version of the library SE.DPAC. It has to be version 23.0.0.10 or more recent.

    • The parameter Runtime Configuration > Communication > Reliable CrossComm > Protocol > Disable Reliable QoS of both device should be set to FALSE.

      NOTE: This parameter can be found in Properties pad of the devices in the Logical Devices editor.

Both hidden IThis variable and function block instance have the same name A

As described in the topic OPC UA Server Editor, this error indicates that because of hiding the IThis from the hierarchy, two elements end up with the same name:

  • One of the IThis function block variable.

  • One of the function block instance positioned at the same level as the IThis function block.

The communication to the connected OPC UA client then cannot be done as it would create a conflict with the OPC UA path of these elements .

Example Case

  1. Create a CAT.

  2. Open its IThis function block.

  3. In the IThis Interface editor, add an input variable named iName.

  4. Save and add the IThis function block into the Function Block Network editor of the CAT.

  5. Add a CAT instance in this same Function Block Network editor and name it iName.

  6. In the OPC UA Server editor, hide the IThis line by clicking on this icon:

    NOTE: Once hidden, the IThis line will be written in italic and the icon will switch to this one:

  7. Save and add the CAT to the system application Main to create a CAT instance from it.

  8. Name this CAT instance CATName.

  9. Map the CAT instance on a device and compile it.

This example generates the following error while compiling: 

Main::CATName: A variable of a hidden CAT_HMI block and a function block instance have the same name iName. This will result in the duplication of OPCUA node id (ERR_CONFLICT_OPCUA_PATH)

In this example, the CAT instance has two elements named identically: the variable iName of the IThis and the function block iName. As the IThis line has been hidden in the OPC UA Server editor, then both these elements with the same name becomes at the same level in the CAT instance which is creating the conflict leading to this error.

Right click in the error and select Show in Editor, this will show where the error is located. Then change either:

  • Change the OPC UA Server parameter from String to GUID. This parameter is accessible with the following path: Solution Explorer > Right click on the application > Options > IEC61499 > Settings.

Compile the solution to check that the error is removed.

ERR_CONST_AMBIGUOUS

(${1},${2}) ${eCls} ${eSym}: Constant '${3}' is ambiguous ('${4}' or '${5}')

The constant named in ${3} is defined multiple times and the constants type cannot be determined unambiguously. The constants must be typed if the constant is defined in differently named types in the same name or namespaces. Constants of the same name in different namespaces do not cause a conflict because looking up unqualified types is defined by the order of USING directives but the order of looking up constant identifiers in different types in the same namespace is not. Therefore, typing a constant always resolves ambiguities, qualified typing is not required unless the type (in absence of a USING directive) cannot be found implicitly.

Example Case
For the following ST test code, Variable1 is a SE.App2StateManagement.PhaseStates type element. An ambiguous test code is as followed: 
IF Variable1 = RunningState THEN 
    // action_following_test
END_IF

Instead the following test code should be used : 

IF Variable1 = SE.App2StateManagement.PhaseStates#RunningState THEN
    // action_following_test
END_IF;

ERR_CONST_TYPE

(${1},${2}) ${eCls} ${eSym}: Constant is not of required type '${3}'

This detected error is given if the value of a constant does not match a specific type or a specific purpose which in turn requires a specific type. Examples:

ERR_CONST_VALUE

Value X is out of range for constants of type Y

The value X has been used in an operation or have been assigned to a constant of type Y. However, the value X or the result of its operation is not included inside the value range of the type Y.

NOTE: The value range of all the types are available in the topic IEC 61131 Data Types.

Example Case 1 

FUNCTION test_ERR_CONST_VALUE
	VAR_OUTPUT
		oValue : BOOL;

	END_VAR

	oValue : TO_BOOL(2);

END_FUNCTION

Example Case 2 

FUNCTION test_ERR_CONST_VALUE
	VAR_INPUT
		iValue : UDINT;

	END_VAR
	VAR_OUTPUT
		oValue : UDINT;

	END_VAR

	oValue := iValue * -5;

END_FUNCTION

These examples generate the following error while compiling: 

In Algorithm 'test_ERR_CONST_VALUE': Value '2' is out of range for constants of type 'BOOL' (ERR_CONST_VALUE)
In Algorithm 'test_ERR_CONST_VALUE': Value '-5' is out of range for constants of type 'UDINT' (ERR_CONST_VALUE)

The first example attempt to convert in boolean the value 2 and assign the result to oValue. However 2 is out or range for the type boolean.

The second example attempts to multiply the variable iValue by -5. As iValue is a type UDINT, thus, beside when iValue is equal to 0, its value is always positive. The result can only be negative, which is out of range for oValue as its type is UDINT too.

TIPS

Right click in the error and select Show in Editor, this will show where the error is located. Then either:

  • Change the type of the variable receiving the value.

  • Modify the operation or the assignation issuing.

  • Change the function called or the value given to it.

ERR_CTRLVAR_TYPE

(${1},${2}) ${eCls} ${eSym}: The control variable of the FOR loop must be of type ANY_INT but is of type '${3}'

The variable controlling a FOR loop must be of type ANY_INT. This is especially true for simple enumeration types because enumeration values cannot be converted to integers and there is no ordering defined on enumeration values.

ERR_DECLORDER

(${1},${2}) ${eCls} ${eSym}: Nonstandard variable declaration order

In structured text syntax internal variables in basic function blocks must be declared before plugs and sockets are. In XML, however, the ordering is the other way around. To allow easy transformation from XML to ST the grammar has been extended to allow both forms. Generally, the compiler will report a detected error when XML ordering is used in an ST function block. The FB type compiler which converts all XML to ST enables compiler feature IntVarOrdering to suppress this detected error and make translation to ST more efficient. The detected error is emitted only if plain ST is compiled and XML declaration ordering is used. In this case the ordering of internal variables must be done before the declaration of plugs and sockets.

ERR_DIMENSIONS

(${1},${2}) ${eCls} ${eSym}: The index expression does not match the arrays dimensions

Generally, this detected error is given if an index operator is applied to a non- array argument or if there are more indexes passed to the index operator than the array has dimensions.

ERR_DIMRANGE

(${1},${2}) ${eCls} ${eSym}: Dimension expression out of range

This detected error is given if the dimension expression (second parameter to functions LOWER_BOUND or UPPER_BOUND) is either less than one or greater than the number of dimensions of the array variable given in parameter one.

ERR_DISABLED_POU

(${1},${2}) ${eCls} ${eSym}: Disabling execution of POU '${3}' yields undefined results.

Disabling part of an expression using the EN/ENO feature leads to undefined results and is therefore forbidden. If the right-hand side of an assignment expression consists solely of a possibly disabled function call, the assignment of the undefined POU result is also suppressed. This is not possible for a general expression where enabled and disabled terms may be intermingled and the flow of control is not well defined. Parameter ${3} names the POU which can make the result of an expression undefined.

This detected error message is generally given for all new IEC 61131 third edition data types if code is generated for an intermediate code version prior to 3.04.

ERR_KEYWORD

(${1},${2}) ${eCls} ${eSym}: Identifier '${3}' is a keyword

The identifier given in parameter ${3} is considered a keyword and cannot be used for any other purpose than the one defined in the IEC 61131/61499 standard. Generally, the compiler requires all keywords to be written in uppercase letters. This is a non-conformant behavior because the standard considers the language to be case insensitive. That is why the compiler marks all identifiers which are keywords by definition but are not written in all uppercase letters as detected error. In this way, programs accepted by the compiler are certainly portable but have the limitation that not all programs written for other compilers will compile correctly. Following is a set of rules on handling keyword detected errors:

  • Structured text syntax defines a set of keywords in upper case notation which cannot be used in structured text for any other purpose. Non-use of this syntax is detected by the parser and will cause a syntax detected error (ERR_SYNTAX). This rule applies to all IEC 61499 basic function blocks as well as all IEC 61131 programs, function blocks and functions.

  • In excess to the keywords defined above the standard IEC 61499 and IEC 61131 define a list of keywords which are either not yet used in the compiler grammar or are not strictly considered keywords (e.g. names of all IEC 61131 library functions). For reasons of compatibility these keywords in general cannot be used in structured text code and result in a detected error (WRN_KEYWORD). When compiler feature AllowKeywords is enabled then this restriction is loosened and only a warning is emitted. It does not make any difference whether the keyword is spelled in all uppercase letters or if a mixed case notation is used.

  • In IEC 61499 service or composite function blocks all use of keywords as variable names results in an informational message (INF_KEYWORD). This is mainly done for compatibility reasons. Nevertheless, it is strongly recommended to stay away from the use of keywords whenever possible.

Keyword IN as defined in the IEC 61499 standard is in this compiler implementation not considered a keyword. IEC 61131 uses this identifier as formal parameter name in many of the predefined library function and IN is not considered a keyword in this standard. Therefore, IN can be freely used as identifier. To avoid incompatibility with other implementation use this with care.

ERR_PARAM_NEGATED

The output parameter of a function cannot be negated

NOT operator cannot be used within the output parameters of a function. The output parameters cannot directly be negated, only its return value can be as shown in the example below.

Example Case 

FUNCTION TEST_ERROR_PARAM_NEGATED:BOOL
	VAR_INPUT
		iValue : DINT;

	END_VAR
	VAR_OUTPUT
		oValue : DWORD;

	END_VAR

	oValue := TO_WORD(iValue);

	TEST_ERROR_PARAM_NEGATED := (iValue MOD 2) = 0;

END_FUNCTION
ALGORITHM CALL_FUNCTION IN ST:
	VAR_INPUT
		QIDint : DINT;

	END_VAR
	VAR_OUTPUT
		QOBool : BOOL;
		QODWord : DWORD;

	END_VAR

	QOBool := NOT TEST_ERROR_PARAM_NEGATED(iValue := QIDint, NOT oValue => QODWord);

END_ALGORITHM

This example generates the following error while compiling: 

In Algorithm 'CALL_FUNCTION': Function output parameters cannot be negated (ERR_PARAM_NEGATED)

In this example, the function TEST_ERROR_PARAM_NEGATED gets with the input parameter iValue a DINT value and convert it into a DWORD. This converted value is sent as output value with the output parameter oValue. Finally the function returns 1 or 0 if iValue is odd or even.

Thus, in the function block CALL_FUNCTION, the function TEST_ERROR_PARAM_NEGATED is called by giving QIDint as input and receiving:

  • in QOBool the negated value of the returned value.

  • in QODWord the negated value of the output value.

The error is coming from the NOT used for negating the output value oValue. The output parameter of a function cannot be directly negated while the return value can.

TIPS

Right click in the error and select Show in Editor, this will show where the error is located. Then remove the NOT operator from the output parameter to remove the error. To have the same result without having error, it is recommended to develop and add variables to make sure the negation is not done in the output parameters of the function, as shown below: 

QOBool := NOT TEST_ERROR_PARAM_NEGATED(iValue := QIDint,oValue => QODWord)
QOWord := NOT QOWord;

Once done, then save and build the solution to check that the error is removed.

The type A is incompatible with the type B

This error indicates that there is in a function a type mismatch. The type A is incompatible with the intended use while the type B was expected.

The intended use could be:

  • A boolean expression in an IF, WHILE or REPEAT statement

  • A single element type expression in a CASE statement

  • A ANY_INT expression in a FOR statement

  • Different types in assignment operation

  • ...

Example Case 1 

FUNCTION ERR_TYPE_MISMATCH:BOOL;
	VAR_INPUT
		iVal1 : INT;
		iVal2 : WORD;

	END_VAR

	ERR_TYPE_MISMATCH := iVal1 = iVal2;

END_FUNCTION

Example Case 2 

TYPE A : ARRAY[0..2] OF DINT;
END_TYPE

FUNCTION ERR_TYPE_MISMATCH:Main.A
	VAR_INPUT
		i : DINT;
		av1 : Main.A;
		av2 : ARRAY[0..5] OF DINT;

	END_VAR

	ERR_TYPE_MISMATCH := MUX(i, av1, av2);

END_FUNCTION

These examples generate the following errors while compiling:

In Algorithm 'Main.A' is incompatible with type 'ARRAY[0..5] OF DINT' (ERR_TYPE_MISMATCH)

It is because the array A has a length of 3 ([0..2]) while the other array has a length of 6 ([0..5]) and that the function MUX only accepts parameters with the same type or with implicit conversion.

ERR_XML_SYMLINK_VALUE

${1},${2} ${eCls} ${eSym} : Illegal direction specification '${3}' for a symbolically linked variable

Direction specification for a symbolically linked IEC 61131 function block variable is either '%I*' indicating that the variable is subscribed to some external data source (gets its value from some symbolic link source) or '%O*' indicating that the variable is published and sent to all subscribing symbolic link destinations upon the end of every execution cycle. All other values are incorrect and not accepted.

ERR_XML_SYMTAB_VERSION

${1},${2} ${eCls} ${eSym} : Cannot set immediate code version from '${3}' to '${4}'

This detected error indicates that the compiler cannot create an intermediate code with a version number given in parameter ${4} because of symbol table restrictions (the intermediate code version is kept in the symbol table so intermediate code version and symbol table version are used synonymously). Both version numbers in parameter ${3} and ${4} are given in the form "vrr" where v is the version number and rr the release number ranging from "00" up to "99". Parameter ${4} always specifies the intermediate code version the compiler was requested to produce (e.g. as specified in the compiler target profile).

  1. In case of a standard compiler run parameter ${3} gives the highest version number the compiler is capable to produce. Obviously this version number must be greater or equal to the intermediate code version number the compiler is expected to produce (the compiler can be instructed to produce backward versions of intermediate code only).

  2. In case of an incremental compiler run parameter ${3} holds the version number of the intermediate code version of the originating symbol table. This version number must exactly match the version number of the intermediate code the compiler is expected to produce as shown in parameter ${4}. Currently intermediate code cannot be upgraded incrementally.
