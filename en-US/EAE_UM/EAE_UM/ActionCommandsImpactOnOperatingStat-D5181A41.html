Action Commands Triggering Operating State Transitions

The following action commands, available in the Deploy and Diagnostic editor, can impact a device’s operating state:

  • Clean

  • Deploy

  • Load Boot Project

  • Reboot Device

  • Restart Runtime

  • Run

  • Stop

Clean

The Deploy > Clean command deletes both the application (program logic) and hardware configuration (I/O and communication) settings from the device execution memory.

 WARNING
UNINTENDED EQUIPMENT OPERATION
Before performing a “Clean” command on a running dPAC, make sure that the resulting immediate interruption of the execution will not have unwanted effects on equipment operation.
Failure to follow these instructions can result in death, serious injury, or equipment damage.

This command can be executed as follows:

  • Manually via Deploy > Clean. The device enters the Clean state.

  • Automatically via:

    • Deploy > Deploy, with the Deploy Service > Clean before deploy parameter set to True. Then the device enters the Stopped state unless the Runtime Configuration > Deploy Settings > Auto start after deploy parameter is set to True, causing the device to enter the Running state.

      NOTE: For devices that include a physical STOP switch, like M251d and M262d, the device enters the Running state only if the switch is set to the RUN position. If the switch is set to STOP, the device remains in the Stopped state.

    • Device actions > Reboot Device, or cycling device power, which clean the device before loading the boot project, then the device enters the Stopped state, unless the BehaviorAfterReboot parameter specifies differently.

When the device is in Clean state, an application can be written to the device execution memory using:

  • Boot Project > Load Boot Project.

  • Deploy > Deploy.

Deploy

The Deploy > Deploy command writes the compiled project in the device execution memory, including both the application (program logic) and hardware configuration (I/O and communication).

This command can be executed in any operating state, but requires the device to transition to Clean state.

Clean state can be achieved:

  • Manually by executing a Deploy > Clean command, or

  • Automatically when executing a Deploy > Deploy command with the Deploy Service > Clean before deploy parameter set to True.

 WARNING
UNINTENDED EQUIPMENT OPERATION
Before performing a “Clean” command on a running dPAC, make sure that the resulting immediate interruption of the execution will not have unwanted effects on equipment operation.
Failure to follow these instructions can result in death, serious injury, or equipment damage.

After a successful deployment, the behavior of the application depends on the parameter (set in the Logical Devices editor)  Runtime Configuration > Deploy Settings > Auto start after deploy. If this parameter is set to:

  • True: the application immediately enters the Running state (unless a physical STOP switch on the device is set to the STOP position, thereby preventing the start of the application).

  • False: (the default setting) the application enters the Stopped state. To start the application, select the command Device actions > Run.

For very large solutions or very slow devices, the time to complete deployment can be long, resulting in a timeout. To reduce the likelihood of a timeout, increase the value of the long timeout setting. To do this, make edits in two places:

  1. Click File to open the backstage, then select Installed Plug-ins EcoRT Plugin, then click  Settings to access and edit the value for Long Command Timeout. 

  2. In the Logical Devices editor, select a device, then in the Properties pad, navigate to Runtime Configuration > Communication > Deploy  then edit the setting for the TimeoutLong parameter.

After a full deploy, inputs and outputs go to their fallback state and the application enters Stopped state.

Load Boot Project

The load boot project command writes in the device execution memory the project previously stored in the device via the Boot Project > Set Running Project as Boot Project command.

The boot project includes both the application (program logic) and hardware configuration (I/O and communication) settings. This command is available only in Clean state.

The resulting operating state can be:

  • Clean: if no coherent boot project exists. A boot project is coherent when all of the following conditions are met:

    • It exists.

    • It is not corrupted and has not been tampered with.

    • It is compatible with the solution's hardware configuration.

  • Stopped:

    • if a coherent boot project exists, and if the Runtime Configuration > Deploy Settings Auto start after deploy parameter = FALSE

    • if a physical STOP switch is STOP

  • Running if a coherent boot project exists, and if the Runtime Configuration > Deploy Settings Auto start after deploy parameter = TRUE, and in the absence of a physical STOP switch in STOP position.

  • ErrorHalt: if an error is detected when parsing the coherent boot project.

NOTE: The boot project will also be loaded upon Restart Runtime, Reboot Device and cycling device power, but the resulting operating state will then depend on the Behavior on reboot parameter.
NOTE: If the boot project itself causes an execution error, the running device will go to Error Halt state repeatedly after each attempt to execute it, creating an endless loop. This could happen in several scenario (after Load Boot Project, Restart Runtime and Reboot Device commands or device power cycle, if the settings trigger the boot project execution).

Follow these steps to avoid such a loop:

  1. Boot Project > Delete Boot Project (causing the error loop).

  2. Deploy > Clean.

  3. Deploy > Deploy (another project).

Reboot Device

The Device actions > Reboot Device command immediately stops the application execution and causes the device to power cycle. The services running on the device, including both the dPAC EcoRT runtime service and any non-EcoRT services, are re-initialized.

Non-EcoRT services running on the device can include (but are not limited to) EcoStruxure Automation Device Maintenance (EADM) server, Network Time Protocol (NTP) client, and the EtherNet/IP communication service.

Use the Reboot Device command (instead of the Restart Runtime command) if non-EcoRT services need to be re-initialized.

Use the Restart Runtime command if only the EcoRT service needs to be re-initialized.

The Reboot Device command impacts the dPAC EcoRT runtime service in the same way as the Restart Runtime command. Refer to Restart Runtime for the details on how the Reboot Device command impacts the potential resulting operating states.

NOTE: The Reboot Device command does not impact the dPAC EcoRT runtime service running in simulation mode in EcoStruxure Automation Expert.

Restart Runtime

The Device actions > Restart Runtime command:

  • Immediately stops the application execution, and

  • Causes the termination and restart of the dPAC process.

The content of the device execution memory is lost, and the device attempts a startup process based on the following logic (identical for Reboot Device and device power cycle):

  1. absence of boot project => Clean

  2. detected error while checking the boot project => Error Halt

  3. (with valid boot project loaded in device execution memory) depending on the value of Runtime Configuration > Boot > Behavior on reboot property:

    • Run: Device enters Running state (unless prevented by a physical STOP switch set to STOP).

    • Stop: Device enters Stopped state.

    • LastState: The most recent state is attempted:

      • Clean => Clean

      • Running or Stopped => Running or Stopped (leveraging a physical STOP switch position)

      • Error Halt

        • if the error was found on a Clean state => Clean

        • if the error was found on a Running state => Stopped

    • LastStateIgnoreErr: The most recent state is attempted:

      • Clean => Clean

      • Running or Stopped => Running or Stopped (leveraging a physical STOP switch position)

      • Error Halt

        • if the error was found on a Clean state => Clean

        • if the error was found on a Running state => Running or Stopped (leveraging a physical STOP switch position)

Run

The Device actions > Run command:

  • Is only available when the device is in the Stopped state either:

    • after execution of the Deploy > Deploy (full deploy) command, or

    • after a Load Boot Project command.

  • is not available after a Device actions > Stop command.

  • and, (providing the absence of a physical STOP switch in the STOP position):

    • triggers the execution of the application stored in the device execution memory.

    • sets the device to the Running state.

Stop

The Device actions > Stop command

  • is available only when the device is in the Running state.

  • immediately stops the application execution.

  • sets the device in the Stopped state.

NOTE: After a Device actions > Stop command, use either a Device actions > Reboot Device or Device actions > Restart Runtime command to move the device from Stopped to Running.
