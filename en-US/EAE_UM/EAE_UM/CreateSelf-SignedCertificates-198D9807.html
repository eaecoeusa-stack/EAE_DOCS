Create Self-Signed Certificates

Perform the following tasks in sequence to create self-signed certificates to secure an MQTT connection:

  • Create a CA Certificate

  • Create a Broker Certificate

  • Create a Client Certificate

  • Configure the MQTT Client

Create a CA Certificate

NOTE: If you already possess a CA certificate (or a commercial CA) you can skip this task and substitute your existing CA certificate for the self-signed CA certificate.

  1. Open the Windows command prompt.

  2. Navigate to a Windows folder of your choice, where you want the certificates to be created.

  3. Generate a private key certificate for the CA (the ca.key), which will be used in signing both the broker's public key certificate and the EcoRT's public key certificate:
    openssl genrsa -des3 -out ca.key 2048

    NOTE:

    • You will be asked to create and verify a pass phrase. You define the pass phrase, which is used later to finalize the creation of the certificates.

    • The "ca.key" file contains the private key. It must be treated as confidential. Anyone who can read its content could use it to sign their own certificates and compromise the security of the MQTT communications.

    You will be asked to create and verify a pass phrase. You define the pass phrase, which is used later to finalize the creation of the certificates.

  4. Generate the public key certificate (the ca.crt) by extracting it from the private key certificate created in the previous step, thereby creating a self-signed certificate for the CA:
    openssl req -new -x509 -days 1826 -key ca.key -out ca.crt

    After entering the command, you will be asked series of questions. Your answers to these questions will be embedded in the CSR. Answer the questions as described below:

    Country Name (2 letter code)

    The two-letter country code where your company is legally located.

    State or Province Name (full name)

    The state/province where your company is legally located.

    Locality Name (e.g., city)

    The city where your company is legally located.

    Organization Name (e.g., company)

    Your company's legally registered name (e.g., YourCompany, Inc.).

    Organizational Unit Name (e.g., section)

    The name of your department within the organization. (You can leave this option blank; simply press Enter.)

    Common Name (e.g., server FQDN)

    Write localhost or the fully-qualified domain name (FQDN) (e.g., www.example.com).

    Email Address

    Your email address. (You can leave this option blank; simply press Enter.)

    A challenge password

    Leave this option blank (simply press Enter).

    An optional company name

    Leave this option blank (simply press Enter).

    Some of the above CSR questions have default values that will be used if you leave the answer blank and press Enter. These default values are pulled from the OpenSSL configuration file located in the OPENSSLDIR. If you want to leave a question blank without using the default value, type a "." (period) and press Enter.

Create a Broker Certificate

  1. Generate a private key certificate (server.key) for the broker:
    openssl genrsa -out server.key 2048

  2. Generate a certificate signing request for the broker (server.csr) by extracting it from the private key certificate created in the previous step.
    openssl req -new -key server.key -out server.csr

    NOTE: If you already possess a CA certificate (or a commercial CA) you can substitute your existing CA certificate for the self-signed CA certificate.

    Following this command, enter again some of the previously requested information, including:

    • A challenge password

    • The company name (optional)

  3. Sign the broker's CSR with the CA's private key to get the broker's public key certificate:


    openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out server.crt -days 360
    NOTE:

    • The last characters -days 360 define the duration of the certificates. When the certificates expire, you need to again perform all the steps of this procedure.

    • The "ca.key" file contains the private key. It must be treated as confidential. Anyone who can read its content could use it to sign their own certificates and compromise the security of the MQTT communications.

Create a Client Certificate

You can authenticate a client in two ways, using either:

  • a username/password combination, or

  • a client certificate

The following procedure describes how to create a client certificate for authentication:

  1. Generate a private key certificate for the EcoRT client:


    openssl genrsa -out client.key 2048
    NOTE:

    • This command requests to enter a pass phrase and to verify it. This pass phrase is defined by you and it is used later to finalize the creation of the certificates.

    • The EcoRT client private key certificate will be used in the MQTT_CONNECTION FB as the CertificateKey value.

  2. Generate a certificate signing request (CSR) for the EcoRT client:
    openssl req -new -key client.key -out client.csr

  3. Sign the EcoRT client's CSR with the CA's private key to generate the EcoRT client's public key certificate:
    openssl x509 -req -in client.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out client.crt -days 360

    NOTE:

    • The last characters -days 360 define the duration of the certificates. When the certificates expire, you must complete again all the steps of this procedure.

    • The "ca.key" file contains the private key. It must be treated as confidential. Anyone who can read its content could use it to sign their own certificates and compromise the security of the MQTT communications.

  4. Install or add the ca.crt public key file to the broker's trusted certificate store, and configure the broker with its public key certificate (and its private key certificate if required).

  5. Install or add the ca.crt, client.key and client.crt files to the EcoRT.

    The location may vary by platform. The ca.crt may need to be added to the Windows or Linux certificate store (e.g., using the add-trusted-cert or similar command).

Handling the Private Key File

The "ca.key" file contains the private key. It must be treated as confidential. Anyone who can read its content could use it to sign their own certificates and or compromise the security of the MQTT communications.

Handle the "ca.key" file as follows:

  • Store the "ca.key" file in a folder under least privilege during signing.

  • Allow the "ca.key" file to reside on the file system only while signing is taking place.

  • Scrub the “ca.key” file from the file system after it is removed for secure storage.

Configure the MQTT Client

  1. Add the following path for storage of client certificates for Windows Soft dPAC:

    C:\ProgramData\Schneider Electric\SE SoftdPAC EcoStruxure Automation Expert 24.0\auth

  2. Pass the certificate files to the Soft dPAC host.

  3. Copy certificates to the Soft dPAC container by executing the following commands on the Soft dPAC Linux host:

    • sudo docker cp ca.crt Softdpac_1:/var/lib/nxtSRT61499N/data/auth

    • sudo docker cp client.crt Softdpac_1:/var/lib/nxtSRT61499N/data/auth

    • sudo docker cp client.key Softdpac_1:/var/lib/nxtSRT61499N/data/auth

    NOTE: Replace Softdpac_1 in the upper commands by the instance name of your Soft dPAC.

  5. Configure the MQTT broker to use certificates:

    • require_certificate: if true, the broker informs the client it needs to provide a certificate.

    • use_identity_as_username: if true, the broker takes the username from the certificate and not from a password file.

    • crlfile: You can create a certificate revocation file listing revoked client certificates.
