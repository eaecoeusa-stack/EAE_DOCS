Tips and Tricks

Detected error messages can be presented by message dialogs or in the Message pad. This topic presents some examples of detected errors, and how to handle them.

color name ... does not exist

If this message dialog detected error message occurs, after opening a project or a graphic editor within a project (CAT symbol, Faceplate, Canvas, Graphic) a named color is missing in the project, which is already used in the specified object.

Solution: Execute a right-click on the project node within the Solution Explorer and select Named Styles > Open from the context menu. Create a new color with the name specified in the detected error message. Then set the properties of the color according to the requirements.

In Algorithm 'REQ': Local variable 'outMONTH' hides variable...

The full message pad error may read as follows:

In Algorithm 'REQ': Local variable 'outMONTH' hides variable 'Main#DAYOFWEEK_DAYOFYEAR_SPLIT.outMONTH' [XML Line: 42, Column: 15] - C:\Users\DemoUser\Documents\Projects\FIDO_PROJEKTE\CONVERT_TO_TOD_etc\IEC61499\DAYOFWEEK_DAYOFYEAR_SPLIT.fbt:42,268435455

If a detected error of this kind is thrown, there are two equally named variables of different kind created within one Basic function block. One variable exists only within the algorithm named in the detected error message and one global variable, which has been created in the Function block Interface Editor. This will lead to wrong computation respectively to the output of wrong values in most cases and therefore it has to be reassessed.

Solution: rename one of the two variables.

  1. Compiler warnings such as:

    Warning : In ECTransition 'START TO INIT': Compatibility warning: event variables are not allowed in a guard condition

    Warning : In ECTransition 'START TO INIT': Compatibility warning: event variables must be ANDed to the guard condition

    Warnings of this kind are shown if an ECC transition condition of a Basic function block does not meet the specified guidelines. A possible reason for such a warning to appear suddenly may also be that the check mark in the setting Allow general Boolean expressions in transition conditions in Options > IEC61499 (after right-clicking the project node in the Solution Explorer) has been removed, although the project was originally created with the tick set. This setting (if false) causes a strict check of Boolean expressions of transition conditions according to the IEC 61499 standard. By default, this setting is True.

Cannot write to output file...

If an error of this kind is thrown, the path file name of the file in question is too long. The restrictions defined by Microsoft Windows are valid.

Solution: The project has to be moved to another folder somewhere higher in the file hierarchy in order to keep the total name of the file shorter, respectively you can choose a shorter name for the project. See also Home menu at item Build Solution.

The required attribute 'busId' is missing...

If a warning of this kind appears, an attribute is missing that has to be added by the use of the Properties dialog.

Solution: Open the dialog by right-clicking the appropriate device in the Hardware Configuration editor and select Properties.... Complete the configuration of the busid  property, then re-deploy. The message should disappear.

DEV0: LogEntry: Error: Stored persistent data do not match this project...

The full message reads: DEV0: LogEntry: Error: Stored persistent data do not match this project, delete it and startup - C:\Users\DemoUser\Documents\Projects\Testproject\IEC61499\System\System.sys

This detected error indicates that the persistent data on the runtime does not match the project that is to be loaded. To delete the persistent data, open the Deploy and Diagnostic editor, and select Action > Delete Persistent Data from the context menu by right-clicking on the corresponding device.

NOTE: Save persistent data using the Fetch Persistent data from the device context menu before selecting Delete Persistent Data. Failure to do so can lead to the loss of persistent data.

In Algorithm 'REQ': NOT operation on untyped integer value 16#1, assuming 64 bit wordsize (LWORD) [...]

The compiler emits a warning like this when using untyped literals. See the note on Implicit Dataconversion.

Which could also lead to a detected error like:

In Algorithm 'REQ': Value '18446744073709551614' is out of range for constants of type 'WORD' [...]

The HMI application cannot be deployed...

The full message looks like this: The HMI application cannot be deployed. Ensure that port '51443' is not filtered by your firewall and HMI manager is reachable with the configured IP Address or that your HMI runtime version is compatible with current version of 'EcoStruxure Automation Expert'.

In this case, you need to:

  • Determine if the port is closed and, if so, open it in the Microsoft Firewall as follows:

    1. In the task bar, right-click HMI Manager and select Settings.

    2. Confirm the Port Number in HMI Manager is the same as the port number in your solution.

    3. If the two port numbers are different, change the port number in HMI Manager Settings to match the port number in the solution.

    4. If the port numbers are the same and the detected error persists, verify the port is open, as follows:

      1. Open a command line interface on the Windows host.

      2. Execute the command:
        netstat -ano | findstr : port number

        For example:
        netstat -ano | findstr : 51443

        If the port is open and listening, it will be displayed in a list of open ports.

    5. If the port does not appear in the list of open ports, the port is blocked by your Microsoft Firewall. In this case, you need to enable the timed groups to open the port.

  • Check HMI runtime compatibility. Refer to the EcoStruxure Automation Expert Compatibility Guide topic EcoStruxure Automation Expert HMI.

  • If there is a compatibility issue, upgrade the HMI Runtime. Refer to the HMI User Guide topic Upgrade EcoStruxure Automation Expert -HMI.
