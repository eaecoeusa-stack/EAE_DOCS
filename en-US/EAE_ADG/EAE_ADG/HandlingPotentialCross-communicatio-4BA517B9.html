Handling Potential Cross-communication Event Loss

IEC61499 standardizes a way to program distributed systems and EcoStruxure Automation Expert allows users to create and manage such distribution transparently, providing automatic generation of the necessary communication between remote function blocks. But this communication protocol does not provide any native acknowledgement feature. Therefore, any requirement to monitor the reception of messages (events) must be implemented at application level.

 WARNING
LOSS OF EVENTS
  • Identify the events on which the safety of the application depends.
  • Always handle these events at application level, implementing algorithms (like event reception acknowledgement and event re-send) to account for the possibility that events could be lost between transmission and reception.
Failure to follow these instructions can result in death, serious injury, or equipment damage.

It is the responsibility of the designer to identify the events that need monitoring.

As a guideline, both the frequency of the firing of the event, and the severity of the consequences of losing the event should be considered.

NOTE: Monitoring of events itself may impair the system performances.

If not received, an event that is rarely fired would leave the system in an inconsistent state for a long period of time (until it is fired again and received). This would be the case for instance for an event sent when the level of water inside a tank crosses a set level. On the contrary, if the event is frequently sent, then a lost event would probably result in a short duration inconsistency. This would be the case if the event is sent every time the water level in the tank changes by 1mm.

NOTE: Most importantly, the potential consequences to the system, the safety of individuals, or potential material damages, in determining whether an event needs to be monitored to confirm it is duly received and acknowledged by the recipients.
