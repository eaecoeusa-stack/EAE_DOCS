Events Management in a Resource

Inside each resource, events are managed in two events queues:

  • The external queue contains external events received by the resource.

  • The internal queue contains internal events generated by the function blocks belonging to the resource.

Each external event is processed only after all internal events generated by the processing of the previous external event are processed themselves.

  1. The external events received by a resource are stored into the external queue and are processed by the execution engine in order of their reception (FIFO queue) as the targeted function block is executed.

    1. The target function block generates output events (in most cases) which are:

      • Pushed to the internal queue if the target is in the same resource (internal events).

      • Sent to another resource through cross communication otherwise (external events).

    2. All the events present in the internal queue will be processed until this queue is empty (i.e., when there will be no more internal events generated following the processing of the external event).

  2. Then the execution engine will resume the processing of the external queue with the next event stored.

Events management:

As a consequence:

  • The more events created by the application logic, the longer the delay is until processing the next external event.

  • If the rate of events creation exceeds the capability of the CPU to execute events in the application logic, then both queues can fill-up, timely execution performance will degrade, and ultimately a queue overflow can trigger a process halt.

 WARNING
performance degradation and delay in processing
Avoid overloading the resources with events. Event management is sequential and might overflow the queue causing delays in event processing, triggering of process halt and / or performance degradation..
Failure to follow these instructions can result in death, serious injury, or equipment damage.

Source of events in IEC61499:

  • I/Os: The I/O’s generate events when their state changes.

  • HMI: The operator triggers an action via the HMI (e.g. Start and Stop machine, update parameters).

  • Internal timers: E.g. E_CYCLE, timers.

  • External application: E.g. communication with Scada, HMI, IT system.
