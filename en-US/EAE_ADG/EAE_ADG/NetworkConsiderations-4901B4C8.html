Network Considerations

Internal Communication Protocol

The communication between controllers is based on a UDP standard point to point communication mechanism (not a broadcast mechanism).

The communication with the HMI is implemented through WebSocket.

Event-driven Communication

Contrary to the historical IEC61131 systems (which are based on cyclic communication and computing), the IEC61499 based EcoRT system implements computing and communication mechanisms based on the discrete occurrence of user defined events. The communication of these events is not natively cyclic: it occurs only once, and there is no native acknowledgment of the reception (“send and forget” principle).

Consequently, special application designs must be implemented by the user when it is important to ensure that events are received - for instance despite communication loss.

 WARNING
LOSS OF EVENTS
  • Identify the events on which the safety of the application depends.
  • Always handle these events at application level, implementing algorithms (like event reception acknowledgement and event re-send) to account for the possibility that events could be lost between transmission and reception.
Failure to follow these instructions can result in death, serious injury, or equipment damage.

Topology of Networks

The topology of the network influences the performance and the quality of the communication (message loss, order of packets, potential firewall etc.…). When using a distributed system network topology is a key element to consider when designing the application.

If you use routers and there are firewalls between your engineering workstation using EcoStruxure Automation Expert and the controllers, then configure the firewalls to allow the following list of essential services.
