Mapping the Logic inside EcoStruxure Automation Expert

The deployment of the application allocates (distributes) parts of the application to the targeted controllers.

The application mapping allows users to choose where to run the different parts of the application. The flexibility is huge.

The mapping can be modified afterwards, and the function blocks can be distributed differently without impacts on the logic.

The necessary cross-communication between the controllers is generated automatically by the tool without any specific developer effort.

Every function block added in an application must subsequently be mapped to an available resource (within a device). The project cannot be loaded to a controller without mapping.

The following rules or constraints apply to mapping to avoid performance issues and to be able to access the I/O.

If the parent object is:

  • A Composite object type (CAT implemented as Composite, Composite, Basic): The developed logic inside a function block is always mapped to the same device as the parent object.

  • A SubApp object type (CAT implemented as SubApp, SubApp): The developed logic inside a function block can be mapped to different devices as the parent object.

In this example the sub function blocks are either:

  1. Mapped all together with the parent function block to one device, if the parent function block is from an object type Composite.

  2. Mapped to different devices, if the parent function block is from an object type SubApp.

TIP: Design and map the application in a way that any controller can drop out and rejoin at any time without causing harmful output. System integrity needs to be tested during commissioning and continuously monitored during operations by means of dedicated function blocks in the application.
