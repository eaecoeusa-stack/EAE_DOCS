Using Multiple Resources with Configurable Priorities

For Soft dPAC, M262d and ATVd controllers, you can create multiple resources, up to three, and configure them with different execution priorities. You can then isolate and prioritize bus masters and assign the time-critical parts of the application to higher prioritized resources. This can:

  • Reduce the frequency of occurrence and the magnitude of jitter.

  • Reduce bus cycle messages sent to EcoRT logs.

  • Reduce I/O going to their fallback positions during CRD online change.

    NOTE: Only available with Soft dPAC.

How to Configure Resource Prioritization

Open the Device topic inside the EcoStruxure Automation Expert User manual and scroll down to the New resource context menu section to see how to configure and prioritize a resource.

Resource Prioritization Minimal Requirements

  • Use the DPAC_FULLINIT composite FB in each resource and start your application logic from the INITO event.

  • In the Hardware Configuration editor, verify that the autoStart parameter is set to FALSE for all bus masters. This requires each bus to be initialized before beginning processing.

  • For each resource, set alignWithResPriority to TRUE (when available at bus master - Modbus TCP, Serial) to align the priority of threads with the resource priority.

    Apply OS optimizations.

Resource Prioritization Configuration Tips

  • Do not use resource execution priorities to sequence execution of your application:

    • The events of a resource may be processed in parallel in multi-core environments.

  • Limit interactions between resources to improve Application Response Time:

    • Even limited intra-device cross-communication carries a cost. Continuously executing back and forth communication between resources (RES0 to RES1, then RES1 to RES0, and so forth) can increase jitter.

    • Cross-communication and symlinks rely on external events. Avoid situations where a high-priority resource must wait for a lower priority resource. Otherwise execution of application logic in the high-priority resource can be delayed by a low priority resource that is busy processing internal events.

  • When possible, map only a single bus master to a resource and apply a unique execution priority to each resource.

    • The higher the time criticality, the higher the resource execution priority.

  • Carefully select application parts to be prioritized to improve Application Response Time.

    • Place an entire application into a high priority resource brings no benefit.

    • Map only time-critical logic (for example: reading an input > followed by limited processing to determine action > then writing to an output) to a higher priority resource.

    • Assign higher priority only to a maximum of 10% of the entire application.
